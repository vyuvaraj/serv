# Serv Language — Critical Analysis & Roadmap

## Executive Summary

Serv has achieved impressive feature breadth for a single-developer language project. The core compilation pipeline works, the runtime is functional, and the language covers its target domain (services, schedulers, pub/sub, APIs) well. Both `language-roadmap.md` and `next-roadmap.md` are ~90% complete.

However, the project has accumulated significant **depth debt** — features exist but lack the robustness needed for production trust. The type system is the central weakness: nearly everything flows through `interface{}`, which cascades into performance issues, weak error messages, and an inability to catch bugs at compile time.

The roadmap below focuses on what will move Serv from "working prototype" to "language developers would choose over writing raw Go."

---

## Critical Analysis

### Strengths

| Area | Assessment |
|------|-----------|
| **Feature velocity** | Extraordinary breadth for the project size. 46 stdlib modules, 6 broker integrations, WebSocket, MCP tools, migrations, TLS — all present. |
| **Developer experience** | REPL, formatter, hot-reload watcher, package manager, LSP — the toolchain is already solid for an alpha language. |
| **Domain fit** | The service-oriented primitives (`route`, `every`, `cron`, `subscribe`, `spawn`) genuinely simplify the boilerplate Go requires for these patterns. |
| **Code generation quality** | Generated Go is readable and debuggable. The `// Code generated by Serv compiler. DO NOT EDIT.` discipline is clean. |
| **Stdlib philosophy** | Writing stdlib in Serv itself is a great forcing function — it proves the language can express real patterns. |

### Weaknesses

#### 1. The `interface{}` Problem (Critical)

The type system is the project's Achilles' heel. Current type inference (`getExpressionType()`) handles literals and a few call patterns, but loses type information through:
- Function calls to untyped functions
- Destructuring
- Collection element access
- Cross-function data flow

**Impact:** Every untyped value generates runtime type-switch closures for arithmetic, comparison, member access, and indexing. A simple `let x = a + b` in an untyped context produces a 10-line IIFE in the generated Go.

**Benchmark reality:** An untyped HTTP handler generates ~8 allocations per request where hand-written Go needs ~1. This is a fundamental competitiveness problem.

#### 2. Error Model Mismatch (High)

`try/catch` is implemented via `panic/recover`, which is:
- Non-idiomatic Go (panics are for unrecoverable errors in the Go ecosystem)
- Performance-costly on the error path
- Incompatible with Go's error-as-value convention
- Invisible in stack traces when things go wrong in production

The language has multi-return (`let val, err = expr`) which is closer to Go idiom, but there's no union type (`T | error`) and no compiler enforcement of error handling.

#### 3. No Compile-Time Safety (High)

The compiler does almost no semantic analysis:
- No type checking (you can pass a string where an int is expected)
- No unreachable code detection
- No unused variable warnings
- No missing return detection
- No interface satisfaction checking

This means Serv's type annotations (`fn add(a: int, b: int) -> int`) are currently **documentation only** — they're emitted in generated Go but the Serv compiler doesn't enforce them.

#### 4. Missing Operators (Medium)

- No modulo (`%`) — confirmed absent from the lexer
- No `break` / `continue` — not in the keyword table
- No bitwise operators (`&`, `|`, `^`, `<<`, `>>`)
- No compound assignment (`+=`, `-=`, `*=`, `/=`)
- No ternary or conditional expression

These are all basic operations that users will hit within their first hour.

#### 5. LSP is Minimal (Medium)

The LSP exists (~450 lines) and provides:
- Diagnostics (parse errors only — no semantic errors)
- Keyword + builtin autocomplete
- Basic hover (symbol on current line only)
- Go-to-definition (same-file, name matching only)
- Document symbols

Missing: cross-file navigation, rename, find references, signature help, formatting integration, code actions, workspace symbols. The hover only matches symbols by line number, so hovering on usage (not definition) shows nothing.

#### 6. Testing Framework is Thin (Medium)

- `assert` checks `v == nil || v == false` — no equality assertions, no error messages
- No test isolation (all tests share global state from `init()` functions)
- No coverage reporting
- No setup/teardown hooks
- No table-driven test support
- No timeout per test

#### 7. Module System Fragility (Low-Medium)

- Relative paths only (`../stdlib/response.srv`) — breaks when files move
- No package registry or version resolution
- No circular dependency detection beyond a visited map (no cycle error)
- `export` doesn't actually restrict visibility in the generated code — it's an AST marker

## Roadmap

### Phase 1: Language Foundations (High Priority)

These are table-stakes items that block adoption.

| # | Item | Effort | Status |
|---|------|--------|--------|
| 1.1 | **Modulo operator** (`%`) | Small | ✅ Done |
| 1.2 | **`break` / `continue`** | Small | ✅ Done |
| 1.3 | **Compound assignment** (`+=`, `-=`, `*=`, `/=`, `%=`) | Small | ✅ Done |
| 1.4 | **Bitwise operators** (`&`, `|`, `^`, `<<`, `>>`) | Small | ✅ Done |
| 1.5 | **Map iteration** (`for key, value in map`) | Medium | ✅ Done |
| 1.6 | **Slice expressions** (`arr[1:3]`) | Medium | ✅ Done |
| 1.7 | **String concatenation with `+`** for typed strings | Small | ✅ Done (via type inference) |

### Phase 2: Type System (Critical Priority)

This is the highest-impact work. Every improvement here cascades into better codegen, better errors, and better performance.

| # | Item | Effort | Status |
|---|------|--------|--------|
| 2.1 | **Type inference through assignments** | Large | ✅ Done — `let x = 5` tracks `x` as `int`, emits typed Go code |
| 2.2 | **Function return type propagation** | Medium | ✅ Done — `let result = add(1, 2)` where `add -> int` tracks `result` as `int` |
| 2.3 | **Collection element type inference** | Medium | ✅ Done — type inference recognizes homogeneous arrays; runtime still uses `[]interface{}` for collection method compat |
| 2.4 | **Struct field type tracking** | Medium | ✅ Done — `user.age` on a typed struct resolves to the field's declared type |
| 2.5 | **Basic type checking** | Large | ✅ Done |
| 2.6 | **Null safety** (`T?` optional types) | Large | ✅ Done — `T?` allows nil, plain `T` errors on nil assignment |
| 2.7 | **Union types** (`T | error`) | Large | ✅ Done — parsed, codegen emits `interface{}`, type checker validates |

### Phase 3: Performance (High Priority)

Directly gated on Phase 2 progress — better types = less runtime overhead.

| # | Item | Effort | Status |
|---|------|--------|--------|

| 3.2 | **Direct field access for typed structs** | Medium | ✅ Done — `user.name` on typed struct emits `user.Name` |
| 3.3 | **Skip `toSlice()` for typed arrays** | Small | ✅ Done — known `[]interface{}` vars use direct `range` |
| 3.4 | **Bounded prepared statement cache** | Small | ✅ Done — LRU eviction with configurable max size |

## What NOT to Do

| Temptation | Why to avoid |
|------------|-------------|
| Add more integrations (gRPC, GraphQL, etc.) | Breadth isn't the bottleneck. Depth is. |
| Rewrite the parser | Pratt parsing works fine. The parser isn't the problem. |
| Add a type checker before finishing type inference | You need inference first to know what types flow where. |
| Build a package registry | Premature without a user base. stdlib + `serv add` is sufficient. |
| Pursue WASM compilation target | Go codegen works. Don't fragment before the language is solid. |

## Metrics for Success

| Milestone | Indicator |
|-----------|-----------|
| **Language is correct** | `serv build` catches type errors that would otherwise be Go compile errors |
| **Language is fast** | Benchmark: typed Serv HTTP handler has ≤2x allocation overhead vs hand-written Go |
| **Language is trustworthy** | Users can run `serv build` and trust that if it compiles, it works (barring logic bugs) |
| **Language is adoptable** | New user can go from install → working API in under 5 minutes with LSP guidance |
